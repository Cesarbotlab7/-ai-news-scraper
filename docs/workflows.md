# 工作流程细则

## 新任务启动

新项目或新功能启动时，默认先输出：
- 涉及文件清单
- 执行步骤
- 并行/串行建议

等用户确认再动手。

- 一次只问一个澄清问题，不要连环追问
- 给方案前先提 2-3 个备选，说清利弊再让用户选

## Plan 进度管理

对话中涉及 `~/.claude/plans/` 下的 plan 文件时：

1. **主动读取**：立即读取该 plan 文件，扫描所有步骤和验证项的完成状态
2. **汇报未完成项**：列出尚未完成的步骤，不遗漏
3. **完成即更新**：每完成 plan 中的一个步骤，立即将对应 checkbox 从 `- [ ]` 改为 `- [x]`
4. **收尾提醒**：涉及 plan 的对话结束前，主动告知当前整体进度（X/Y 步已完成）和剩余项

## TDD 完整六步流程

1. **先写测试**：在实现之前，先在对应测试文件里写测试用例
2. **覆盖三类场景**：正常路径、边界情况（空值、极值、越界）、错误处理
3. **跑测试确认红**：运行测试，确认因为实现不存在或不完整而失败
4. **写最简实现**：只写刚好让测试通过的代码，不做超前设计或过度抽象
5. **跑测试确认绿**：所有相关测试通过
6. **重构**：在测试保护下优化命名、结构、重复代码

不需要 TDD 的情况：UI 样式调整 / 布局微调 / 一次性脚本 / 探索性代码 / 简单胶水代码 / 配置文件 / typo 修复。

## 并行开发

项目有独立子模块时，Plan 阶段主动规划 Agent 分工边界，写入项目 CLAUDE.md：
- 明确哪些任务可并行（目录不重叠、无数据依赖）
- 明确哪些必须串行（跨目录依赖、共享配置文件改动）
- 根目录文件（CLAUDE.md、.github/）只由人工修改，Agent 不得触碰
